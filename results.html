Result 01 — memory traffic

Traffic falls — but only where operands get reused

Hot-loop bytes read, posit vs f32, per kernel. Matmul-class kernels hit the textbook 2× / 4× floor because each operand is converted once and reused O(N) times. Single-pass kernels barely move — and two of them (cosine_sim, layernorm) move more bytes, because converting f32→posit costs more than the narrow loads save when nothing is reused.

posit16 posit8 1× break-even · 2× · 4× floors

Reading it: bars are × reduction vs naive f32 (longer is better). Reuse-heavy bars sit above 2×/4× because the quire also drops the per-iteration accumulator reload that naive f32 pays; the pure operand-narrowing floor is exactly 2×/4×, which pre-narrowed storage delivers (gemm 128³: 16.78 MB → 8.39 / 4.19 MB). Bars below the 1× line are kernels where in-kernel conversion backfired — the amortization caveat, made visible.

Result 02 — cycles under a memory hierarchy

The cycle win is a cache-fit story

Speedup vs f32 on the sized kernels (32 KiB L1, cycle-accurate). The standout is gemm at 128³: the narrow posit panel fits in L1 where the f32 panel can't, so posit16 runs 6.3× and posit8 9.2× faster. Push to 256³ and the win evaporates — strided access makes the miss count width-invariant. Sequential attention keeps a steady edge that grows with size.

posit16 posit8 1× = no change

Caveat: the small ideal-memory kernels (no size suffix) run cycle-neutral to slightly slower — the cycle win requires both the memory hierarchy and a working set that crosses the cache boundary. And the magnitude (6–9×) is amplified by Rev's edge model — no L2, no prefetcher — so f32 thrashes maximally (237 cyc/MAC); a real L2 + prefetch would shrink the gap. The direction holds; the size does not transfer literally.

Result 03 — accuracy

Four formats, nine decades of error

Max relative error on a log axis. posit16 holds three to four digits; the posit32+quire oracle sits at or left of f32 (exact accumulation kills the √K drift); posit8 lands far right — usable as a bandwidth floor, never as an accuracy target. conv2d posit8 (142×) runs clean off the chart.

Result 04 — the amortization crossover

Where you convert decides whether you win

Same streaming dot product, two strategies. Pre-narrow the data once and stream it (the realistic deployment) and traffic drops exactly 2×/4× while cycles drop too. Convert inside a single-pass kernel and you move more bytes than f32 and run 2.4× slower. The pre-narrowed GEMM split shows why reuse rescues it: conversion is a rounding error against the hot loop.

Why it matters

The thesis claim is bandwidth, and bandwidth only converts to speed when the format is narrowed at rest and reused — exactly the case the cost model is built to detect, and the case the in-kernel rewrite cannot create on a single-pass kernel.
