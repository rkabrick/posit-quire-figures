Methodology — what a cycle number means here

How cycles are measured, and where the to-and-back is charged

Every number on this page comes from hardware counters on the Rev SST simulator (memHierarchy memory model, cycle-accurate core). The harness brackets the kernel call with a pair of custom ecalls that gate Rev's per-kernel counters — cycles and bytes cover only the bracketed region: no startup, no input generation, no result checking.

The f32→posit→f32 round trip: three regimes, all measured

1 · in-kernel conversion (default rewrite)inside the bracket — fully charged. Amortizes with reuse: on gemm 128³ the conversion is 1.3% (posit16) / 1.7% (posit8) of compute, measured with separate conversion-only bracket variants.
2 · single-pass kernelscharged conversion dominates and posits lose — in-kernel conversion runs ~2.3× slower than f32 on the streaming kernel (Result 04). Exactly the case the cost model refuses.
3 · output sidethe final quire rounding and posit→f32 conversion are always inside the bracket, in every baseline.

Why you can trust the ratios

The full suite was re-simulated under two different Rev core timing models (the June composite-latency model and July's pipelined multi-cycle model). Traffic and accuracy are bit-identical across both; every cycle-ratio direction holds, magnitudes shift under ~20% on reuse-heavy kernels. Sensitivity to the timing model is measured, not assumed. posit16 and posit8 appear in every chart, the alternatives comparison (Result 05), and all 174 rows of the table below — including the rows where they lose.

Result 01 — memory traffic

Traffic falls — but only where operands get reused

Hot-loop bytes read, posit vs f32, per kernel. Matmul-class kernels hit the textbook 2× / 4× floor because each operand is converted once and reused O(N) times. Single-pass kernels barely move — and two of them (cosine_sim, layernorm) move more bytes, because converting f32→posit costs more than the narrow loads save when nothing is reused.

posit16 posit8 1× break-even · 2× · 4× floors

Reading it: bars are × reduction vs naive f32 (longer is better). Reuse-heavy bars sit above 2×/4× because the quire also drops the per-iteration accumulator reload that naive f32 pays; the pure operand-narrowing floor is exactly 2×/4×, which pre-narrowed storage delivers (gemm 128³: 16.78 MB → 8.39 / 4.19 MB). Bars below the 1× line are kernels where in-kernel conversion backfired — the amortization caveat, made visible.

Result 02 — cycles under a memory hierarchy

The cycle win is a cache-fit story

Speedup vs f32 on the sized kernels (32 KiB L1, cycle-accurate). The standout is gemm at 128³: the narrow posit panel fits in L1 where the f32 panel can't, so posit16 runs 5.5× and posit8 7.5× faster. Push to 256³ and the untiled win evaporates — strided access makes the miss count width-invariant (tiling fixes it; Result 06). Sequential attention keeps a steady edge that grows with size.

posit16 posit8 1× = no change

Caveat: the small ideal-memory kernels (no size suffix) run cycle-neutral to slightly slower — the cycle win requires both the memory hierarchy and a working set that crosses the cache boundary. And the magnitude (6–9×) is amplified by Rev's edge model — no L2, no prefetcher — so f32 thrashes maximally; a real L2 + prefetch would shrink the gap. The direction holds; the size does not transfer literally.

Where you convert decides whether you win

Why it matters

The thesis claim is bandwidth, and bandwidth only converts to speed when the format is narrowed at rest and reused — exactly the case the cost model is built to detect, and the case the in-kernel rewrite cannot create on a single-pass kernel.

Result 06 — new since June · tiling × narrowing, decomposed

With locality equalized, narrowing buys traffic at cycle parity

transform-dialect linalg tiling (32³) composed with the automatic rewrite — plus a packed-f32 control that isolates each effect. gemm 256³, cycle-accurate memory hierarchy, speedup vs untiled f32:

The four-way split

f32 tiled (unpacked)3.4×
f32 tiled + packed6.5×
posit16 tiled6.7× · 1.9× less traffic than packed f32
posit8 tiled6.7× · 3.5× less traffic

What it means

The cycle win in the cache-resident regime comes from tiling + packing; narrowing then holds cycle parity while cutting the remaining traffic 1.9×/3.5× (the energy lever) and adding the quire's exactness. The packing f32 needs a dedicated pass for comes free with the conversion scratch — at 512³, unpacked f32 tiling fails outright (1.04×, strided tiles alias at the 2 KB row stride) while posit16-tiled holds 6.5×. Tiled accuracy also improves: per-tile rounding at K=32 gives 1.7e-4 vs 9.9e-4 untiled.

Honest note

Softmax pushes values away from unity into posit's taper region, so the near-unity precision advantage narrows here — the quire and traffic advantages are what remain. That is the honest system-level shape of the claim: exact accumulation and 2× fewer bytes in one format, matching two-format mixed precision without the second format.
