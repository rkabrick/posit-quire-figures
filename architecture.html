The system, end to end

Five stages from a kernel to a cycle count

One Bash driver (bench.sh) walks a kernel through the compiler and the RISC-V toolchain; a second (run_rev.sh) runs it on the simulator and parses the statistics back into one row of a CSV.

Source & workloads

Kernels as linalg.generic MAC idioms, each paired with a C harness that seeds inputs, computes an independent f64 reference, and brackets the hot loop. Plus hand-authored posit kernels and two pointed accuracy probes.

gemm · attention_qk · conv2d · mlp · layernorm gemm_pre{8,16}[cv] · attn_pre* (prep vs hot) dotprod_ortho · dotprod_illcond (quire smoking gun) dot_stream (amortization crossover)

posit-opt MLIR / LLVM 21

RISC-V backend

Standard lowerings to the LLVM dialect, translate to LLVM IR, compile and statically link against the bare-metal harness. Out comes a self-contained ELF carrying the custom Xposit words inline.

The dialect

Two types, seven ops, traits chosen on purpose

A standalone domain dialect that interoperates with arith, linalg, scf, and memref — never tagging or forking arith. Validity rules that TableGen can't express (cross-parameter, cross-operand) live in C++ verifiers.

TypeParametersWidthsRole
posit<W, es>width, exponent size8 / 16 / 32Tapered-precision real. Carries MemRefElementTypeInterface so it can be the element type of a narrow scratch buffer.
quire<16·W>width128 / 256 / 512Exact fixed-point MAC accumulator (Posit-2022 16×W rule). Verified at the op level, since a quire alone can't know its posit width.
OpSignatureTraits
quire_init() → quirePure
quire_mac(quire, posit, posit) → quireCommutative · verifier 16·W
quire_extract(quire) → positverifier 16·W
from_f32 / to_f32f32 ↔ positPure
add / mul(posit, posit) → positPure · Commutative

quire_init is not Pure

If it were, CSE would merge two independent inits into a single qclr — leaving the second accumulation running on stale quire state.

quire_mac is not Commutative

MLIR has no partial commutativity. Marking it would license the rewriter to swap the accumulator with a multiplicand. So only add and mul get it.

quire lowers to i1

A dummy SSA token threads ordering through scf.for; the real 512-bit state lives in the implicit hardware register. Every Xposit op is marked side-effecting to block reorder, CSE, and DCE.

The four passes

Annotate, rewrite, mirror, lower

The cost model and rewrite share one hand-written matcher (MACPatternUtils.h): exactly one mulf, one addf that consumes it, the other addend a block-argument accumulator. The narrow-float baseline reuses the same matcher so it is measured under an identical transform.

runs first · annotation

① posit-cost-model

Walks each candidate and writes a posit.skip attribute when a quire rewrite would not pay off. Profitability is a stack of independent, configurable gates:

  • min-trip-count — reduction too short (default 8)
  • max-reduction-dim — K so large input quantization dominates
  • accuracy-target — skip if 2·2^−frac·√K exceeds tolerance
  • max-total-conversions — 2·parallel·reduction volume cap
  • max-parallel-depth — register-pressure guard
  • cache-size-bytes — operand footprint over L1
  • max-quire-regions — function-level serialization on the single quire register
the core rewrite

② linalg-to-posit

A greedy pattern (types unchanged: the function still takes/returns f32 memrefs). Parallel dims become an outer scf.for nest; the reduction becomes an inner loop carrying the quire as an iter_arg; pre-existing C is added back once after extract.

  • mode=quire — exact quire MAC (default)
  • mode=scalar — posit mul/add, isolates the encoding from exact accumulation
  • mode=f32-hoisted — f32 math, same loop shape; isolates hoisting from narrowing
  • posit8/16 → narrow scratch buffers, pre-converted once for cheap hot-loop loads
methodological mirror

③ linalg-to-narrowfloat

Same MAC idiom, but lowered onto IEEE narrow storage so the baselines are honest. Storage is integer-typed (i8/i16); decode is inline bit manipulation, no libcalls in the hot loop.

  • format = f16 · bf16 · f8e4m3 · f8e5m2
  • accum = f32 (mixed precision) or native (accuracy-only)
  • fp8 emitters use round-to-odd scaling for exact RNE
runs last · dialect conversion

④ lower-posit-to-xposit

Converts posit ops to the LLVM dialect as .word-encoded inline assembly. SCF structural conversions propagate the quire iter_arg type change through loop signatures.

  • custom-0 opcode 0x0B; funct7 = (op5<<2) | fmt
  • f32 ↔ posit via hardware pcvt — no SoftPosit calls
  • fixed scratch regs f28/f29/f30 declared as clobbers
  • replaced an earlier SoftPosit llvm.call backend, now removed

Where it stands — three measured wins

Measured on Rev SST, not projected

The current proposal narrative (June) rests on three results that the data actually supports — each reproducible from the kernels and baselines above.

WIN 01 — categorical

Quire exactness

2.0
true dot only the quire recovers

On Gustafson's orthogonal-vector probe, fp32 and even naive fp64 return 0.0; only posit32 + quire recovers the answer. On the ill-conditioned sweep, fp32 fails at κ≈1e8 and fp64 at 1e16, while posit32+quire (4 B) and posit16+quire (2 B) track the exact value throughout — fp64-grade accuracy at half or a quarter of the bytes.

WIN 02 — bandwidth → cycles

Narrow storage, exact accumulation

2× / 4×
hot-loop traffic cut (posit16 / posit8)

On reuse-heavy kernels the cut is textbook-exact and turns into cycles: gemm_n128 posit16 ran 7.5× faster once its panel fit the 32 KiB L1. The win grows with memory latency. The honest caveat is the amortization crossover — converting f32→posit inside a single-pass kernel moves more bytes and runs ~2.3× slower.

WIN 03 — the 2-byte Pareto

posit16 + quire vs fp16 / bf16

~8×
more accurate at equal byte budget

Among two-byte formats, posit16 with an exact quire is the most accurate single-format option — it beats fp16 and bf16 with a separate f32 accumulator, without needing a second accumulator format. This is the cleanest defensible claim and the one the data audit endorsed as the headline.

Honest state — corrections in flight

What the June audit changed

The data is sound; some of the narrative around it was not. These are open items between what the documents claim and what the simulator actually ran.

metadata

posit8 exponent size is inconsistent. Rev simulates posit8 at es=0, but the IR/bench flags pass es=1 and one deck curve assumes es=2. Recommendation: standardize on the draft-standard 8:0 / 16:1 / 32:2. No effect on the numbers — they were all es=0 — but the text must match.

metadata

Quire widths described ≠ widths simulated. The dialect and docs claim the Posit-2022 16×W rule (128/256/512), but Rev runs the SoftPosit draft widths — posit8 quire is 32 bits, not 128. No numerical consequence; the prose describes hardware that didn't produce the data.

retract

"Preserves f32 accuracy" is dropped. posit16 is three to four decimal orders worse than f32. The surviving, stronger claim is Win 03: posit16+quire matches or beats bf16 + f32-accum at the same byte budget.

reframe

posit8 is a bandwidth-floor probe, not an accuracy format. Its K=512 "error" of 0.541 is es=0 range saturation (maxpos≈64), not a measurement. It also anchors the cost model's negative case — the accuracy-target gate correctly refuses every posit8 rewrite.

scope

Bandwidth wins are conditional on reuse. They hold for gemm, attention, conv2d (O(N) operand reuse). On single-pass kernels the in-kernel conversion inverts the win — cosine_sim ~2.3× worse, layernorm ~1.7× worse, mlp slightly worse. The pre-narrowed kernels exist to make this split measurable.

cleared

Arithmetic and lowering are correct. The audit explicitly absolves the Rev posit arithmetic and the dialect lowering — every anomaly traced to format-metadata mismatch, a plotting error, or honest physics that was mis-narrated, never a miscompile.
